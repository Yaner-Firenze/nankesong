# 南客松 S2 身份证绑定与团队成员采集设计

- 日期：2026-03-29
- 状态：Draft for Review
- 适用范围：在现有“后台发卡模式”上新增身份证绑定与团队成员信息采集

## 1. 背景

当前系统已经切换为“主办方后台发卡”模式：公开侧只展示说明与直通卡详情，所有新建动作由主办方在 `/admin/new` 完成。

客户新增要求：

1. 个人直通卡需要与身份证号码唯一绑定
2. 团队直通卡需要由队长填写到场成员信息
3. 团队成员数量为 2～5 人
4. 团队成员信息至少包含姓名与身份证号码
5. 身份证号只在后台可见，不在公开二维码详情页展示

本次改造重点不是改变产品流，而是在后台建卡流程中增加身份证与成员采集能力，同时控制敏感信息的展示边界。

## 2. 目标

### 2.1 主目标

- 为个人直通卡新增身份证号码，并做唯一绑定
- 为团队直通卡新增项目编号
- 为团队直通卡新增 2～5 名成员的姓名与身份证号码采集
- 公开页不展示任何身份证号码
- 后台详情页可完整查看身份证号码

### 2.2 非目标

- 本次不采集身份证照片或扫描件
- 本次不接入公安/实名校验服务
- 本次不新增身份证脱敏展示逻辑到公开页
- 本次不做批量导入

## 3. 数据结构

采用“单表 + JSON 成员列表”方案。

### 3.1 `passes` 主表新增字段

- `identityNumber`
  - 仅个人直通卡使用
  - 存储身份证号码
  - 做唯一约束
  - 旧记录允许为空

- `projectCode`
  - 仅团队直通卡使用
  - 存储项目编号
  - 项目编号与项目名并存
  - 旧记录允许为空

- `members`
  - 仅团队直通卡使用
  - 使用 JSON 存储成员数组
  - 每个成员结构：
    - `name`
    - `identityNumber`
  - 旧记录允许为空

### 3.2 主键规则

- 数据库主键继续使用随机 `id`
- 不使用身份证号码作为数据库主键
- 个人身份证号只作为唯一绑定字段存在

## 4. 页面与表单

## 4.1 后台新建页 `/admin/new`

### 个人直通卡字段

- 姓名
- 联系方式
- 身份证号码
- 项目名称
- 角色
- 项目一句话介绍
- 补充说明

### 团队直通卡字段

- 团队名称
- 主联系人姓名
- 联系方式
- 项目编号
- 项目名称
- 团队人数
- 角色
- 项目一句话介绍
- 补充说明
- 团队成员信息（2～5 组）
  - 成员姓名
  - 身份证号码

## 4.2 后台详情页 `/admin/pass/[id]`

新增后台可见字段：

- 个人卡：
  - 身份证号码

- 团队卡：
  - 项目编号
  - 成员列表
  - 每位成员的身份证号码

## 4.3 公开详情页 `/pass/[id]`

保持当前公开展示逻辑，但：

- 不展示个人身份证号码
- 不展示团队成员身份证号码
- 可继续展示团队人数
- 如需展示成员信息，后续另行决定；本次不在公开页展示成员明细

## 5. 校验规则

### 5.1 个人直通卡

- `identityNumber` 必填
- 身份证号格式必须通过基础校验
- 身份证号必须唯一

### 5.2 团队直通卡

- `projectCode` 必填
- `projectName` 必填
- `members` 数量必须在 2～5 之间
- 每位成员都必须填写：
  - `name`
  - `identityNumber`
- 同一团队内成员身份证号不能重复
- 团队人数与成员列表长度必须一致

## 6. 兼容策略

- 历史记录不强制补录身份证号或成员信息
- 旧记录保持可查看、可搜索、可软删除
- 新建记录按新规则执行
- 数据库新增字段允许旧记录为空，避免线上历史数据报错

## 7. 安全边界

- 身份证号码仅在后台可见
- 二维码内容仍然只编码公开详情页 URL
- 公开页绝不渲染身份证号码
- 不把身份证号作为 URL、二维码内容或数据库主键

## 8. 测试重点

1. 个人直通卡必须填写身份证号
2. 同一身份证号不能重复创建个人直通卡
3. 团队直通卡必须填写项目编号
4. 团队成员数量少于 2 或大于 5 时必须报错
5. 团队人数与成员列表长度不一致时必须报错
6. 团队成员身份证号重复时必须报错
7. 后台详情页可看到身份证号
8. 公开详情页不显示任何身份证号

4. 修改创建 action 与数据映射
5. 修改后台详情页展示
6. 验证公开页不泄露身份证号
7. 补单测与 e2e
