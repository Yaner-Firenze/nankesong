# 南客松 S2 后台发卡模式设计

- 日期：2026-03-29
- 状态：Draft for Review
- 适用范围：替代“选手自助登记生成直通卡”的旧链路

## 1. 背景

当前系统最初按“选手自行填写信息并生成直通卡”的方式设计。虽然主办方仍可在后续环节审核，但公开入口会给外部用户造成“任何人都能自己生成直通卡”的感知。这会模糊直通资格的发放权，也不利于赛事品牌与组织秩序。

客户反馈的目标链路更合理：

1. 主办方先通过飞书表单收集候选个人 / 团队信息
2. 主办方内部确认直通资格
3. 主办方在后台录入系统并生成二维码
4. 主办方将二维码发给对应个人 / 团队
5. 对方在飞书报名表单中提交该二维码

本次改造的核心不是换技术栈，而是将系统定位从“公开登记入口”切换为“主办方后台发卡工具”。

## 2. 目标

### 2.1 主目标

- 收回直通卡创建权限，只允许主办方后台创建
- 公开侧不再承担申请 / 登记职责
- 保留二维码详情展示能力，供已获得直通卡的对象查看
- 保留现有 admin 数据管理能力，并继续扩展为后台发卡工作台

### 2.2 非目标

- 本次不接飞书 API
- 本次不做批量导入
- 本次不做多人账号体系
- 本次不新增复杂审核流，主办方在系统外确认资格后，后台创建即生效

## 3. 用户与权限

### 3.1 公开用户

- 可以访问首页说明页
- 可以访问已发出的直通卡详情页
- 不能创建直通卡
- 不能进入后台

### 3.2 主办方管理员

- 登录 admin 后台
- 查看记录列表
- 搜索记录
- 查看详情
- 编辑内部备注
- 新建直通卡
- 软删除直通卡

## 4. 新链路

### 4.1 主办方链路

1. 主办方通过飞书或其他渠道收集直通对象信息
2. 主办方内部确认资格
3. 主办方登录后台
4. 在 `/admin/new` 中创建个人或团队直通卡
5. 系统立即生成记录并落到后台详情页
6. 主办方复制公开链接或保存二维码并发送给对应对象

### 4.2 选手链路

1. 选手收到主办方发出的直通卡
2. 选手扫码查看直通卡内容
3. 选手在飞书报名表单中提交该二维码 / 链接

## 5. 信息架构

### 5.1 保留页面

- `/`
  - 从“登记入口”改为“直通卡说明页”
- `/pass/[id]`
  - 继续作为二维码实际指向的公开详情页
- `/admin`
  - 后台列表页
- `/admin/pass/[id]`
  - 后台详情页
- `/admin/login`
  - 后台登录页

### 5.2 新增页面

- `/admin/new`
  - 后台新建直通卡页面

### 5.3 下线页面

- `/apply`
  - 直接下线，不再对外提供登记能力

### 5.4 兼容页面

- `/pass/[id]/created`
  - 不再作为主流程使用
  - 可以短期保留做兼容，后续视需要移除或跳转

## 6. 页面设计

### 6.1 首页 `/`

首页改成纯说明页，不再提供“个人登记 / 团队登记”按钮。

建议内容：

- 标题：南客松 S2 直通卡
- 说明：
  - 直通卡由主办方发放
  - 获得直通卡后，请在飞书报名表单中提交
  - 扫码后可以查看对应直通卡内容

### 6.2 后台列表 `/admin`

保留现有能力，并新增“新建直通卡”入口。

功能：

- 搜索记录
- 查看详情
- 从列表进入详情页
- 前往 `/admin/new`

### 6.3 后台新建页 `/admin/new`

用于主办方手动录入信息并创建直通卡。

功能：

- 选择卡类型：个人 / 团队
- 录入字段
- 创建即生效
- 成功后跳转到 `/admin/pass/[id]`

### 6.4 后台详情页 `/admin/pass/[id]`

保留现有详情查看能力，并作为创建后的第一落点。

功能：

- 查看完整信息
- 编辑内部备注
- 软删除记录
- 查看公开详情页对应信息

### 6.5 公开详情页 `/pass/[id]`

继续作为二维码对应页面，但文案改为“主办方发放的直通卡”语境。

展示内容：

- 状态
- 卡类型
- 团队 / 个人信息
- 项目信息
- 飞书报名提交说明

## 7. 字段设计

继续沿用当前系统的主表字段，减少迁移成本。

### 7.1 个人直通卡

- 姓名
- 联系方式
- 项目名称
- 角色
- 项目一句话介绍
- 备注 / 补充说明

### 7.2 团队直通卡

- 团队名称
- 主联系人姓名
- 联系方式
- 团队人数
- 项目名称
- 角色
- 项目一句话介绍
- 备注 / 补充说明

### 7.3 字段语义调整

数据库中的 `userNote` 可继续保留字段名不变，但在后台 UI 中改成“补充说明”或“发卡备注”，避免继续暗示这是用户本人填写的内容。

## 8. 状态规则

本次保持“创建即生效”。

理由：

- 资格确认已经在系统外完成
- 系统内不需要再加一道发布流
- 可降低主办方发卡的操作成本

删除仍采用软删除：

- `status = deleted`
- 后台列表隐藏
- 公开页不可访问
- 数据库记录保留

## 9. 数据与兼容

### 9.1 保留

- `passes` 主表
- 备注、限流、幂等等现有机制
- 已生成的历史二维码链接

### 9.2 不需要迁移的数据

历史记录无需迁移。已有二维码继续访问 `/pass/[id]`。

### 9.3 兼容策略

- 新流程不再使用公开申请页面
- 对旧链接不做破坏性处理

## 10. 实现边界

### 10.1 本次实现

- 首页改版为说明页
- 去掉公开申请入口
- 下线 `/apply`
- 新增 `/admin/new`
- 复用现有表单能力到后台创建流程
- 创建后跳转后台详情页
- 调整公开详情页文案

### 10.2 本次不实现

- 飞书自动同步
- 批量导入
- 多管理员体系
- 系统内审核流

## 11. 测试重点

1. 首页不再提供公开创建入口
2. `/apply` 已不可作为创建入口使用
3. admin 可以进入 `/admin/new`
4. admin 可以创建个人直通卡
5. admin 可以创建团队直通卡
6. 创建后跳转到后台详情页
7. 公开详情页仍然能通过二维码访问
8. 删除后记录从后台列表消失，公开页失效

## 12. 推荐实施顺序

1. 关闭公开创建入口
2. 新增后台创建页
3. 复用现有表单到后台
4. 调整创建完成后的跳转链路
5. 调整公开说明文案
6. 跑全量回归测试

## 13. 结论

本次改造的核心是重新定义系统角色：

- 公开侧：说明与展示
- 后台侧：创建、管理、发放

这样可以把直通资格的发放权清晰地收回到主办方侧，同时保留现有系统中已经成熟的二维码展示和后台记录管理能力。
