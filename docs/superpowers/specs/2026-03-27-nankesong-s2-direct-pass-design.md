# 南客松 S2 直通卡系统设计

## 1. 背景

南客松 S2 需要向部分个人或团队发放“直通卡”。活动方希望给对方一个入口二维码，对方扫码后填写信息，系统再生成一张唯一的第二二维码。活动现场或审核环节扫描第二二维码时，可以直接看到这张直通卡对应的是谁，从而辅助人工放行。

当前约束如下：

- 部署优先选择 `Vercel`
- 先按免费方案启动，默认采用 `Vercel Hobby + Neon Free`
- 不使用 `Supabase`
- 不使用 `SQLite`
- 报名系统本身不需要打通，现场是人工扫描和人工审核
- `admin` 页面只需要简单密码登录

## 2. 目标

这个系统的目标是提供一套轻量但完整的直通卡登记和展示流程：

- 用户通过统一入口二维码进入系统
- 用户先选择直通卡类型：`个人` 或 `团队`
- 用户提交表单后，系统生成一条唯一记录
- 系统返回第二个二维码
- 活动方扫描第二个二维码后，可以看到该个人或团队的核心信息
- 管理员可以在后台查看记录并写内部备注

## 3. 非目标

本期不做以下内容：

- 不与飞书表单做自动同步
- 不做复杂账号体系
- 不做多角色权限管理
- 不做核销状态、签到状态或使用次数统计
- 不做 Excel 导出
- 不做短信或邮件通知
- 不做图片上传或附件上传

## 4. 核心方案

技术方案采用：

- 前端与后端：`Next.js`
- 部署：`Vercel`
- 数据库：`Neon Postgres`
- 认证：基于环境变量密码的简单 `admin` 登录

二维码方案采用：

- 第一个二维码固定指向系统入口页
- 第二个二维码只编码公开详情页 URL
- 二维码中不直接存放用户原始信息

这样做的原因：

- 二维码内容更短，扫描更稳定
- 后续页面展示字段可调整，不影响已生成二维码
- 内部备注不会暴露在二维码内容中
- 公开详情页可长期作为人工审核依据

## 5. 用户流程

### 5.1 外部用户流程

1. 用户扫描第一个入口二维码
2. 进入首页，选择 `个人直通卡` 或 `团队直通卡`
3. 进入对应表单页并提交信息
4. 系统创建一条记录并生成唯一 ID
5. 页面展示提交成功信息和第二个二维码
6. 用户保存第二个二维码，用于后续审核

### 5.2 审核人员流程

1. 审核人员扫描第二个二维码
2. 浏览器打开该记录的公开详情页
3. 页面展示该个人或团队的核心信息
4. 审核人员据此进行人工判断和放行

### 5.3 管理员流程

1. 管理员访问 `/admin/login`
2. 输入系统密码后进入后台
3. 在列表中搜索或浏览记录
4. 查看某条记录详情
5. 填写或更新 `internalNote`

## 6. 页面设计

### 6.1 首页 `/`

作用：

- 作为第一个二维码的落点
- 解释这是“南客松 S2 直通卡登记入口”
- 让用户选择卡类型

页面内容：

- 简短说明
- `个人直通卡` 按钮
- `团队直通卡` 按钮

### 6.2 登记页 `/apply`

作用：

- 根据卡类型展示不同字段
- 完成表单提交

交互规则：

- 页面通过查询参数或路由参数识别卡类型
- 未选择卡类型直接访问时，跳回首页
- 表单提交成功后跳转到创建成功页

### 6.3 创建成功页 `/pass/[id]/created`

作用：

- 告知用户提交已成功
- 展示第二个二维码
- 展示唯一 ID 和详情页链接

页面内容：

- 成功提示
- 唯一 ID
- 二维码图片
- 对应详情页 URL

### 6.4 公开详情页 `/pass/[id]`

作用：

- 供审核人员扫描第二个二维码后查看

展示内容：

- 卡类型
- 状态：`有效`
- 唯一 ID
- 姓名或团队名称
- 主联系人
- 联系方式
- 项目名称
- 角色
- 团队人数
- 项目一句话介绍
- 提交时间

说明：

- `internalNote` 不在公开页展示
- 页面默认只读

### 6.5 管理后台 `/admin`

作用：

- 内部查看和补充信息

MVP 功能：

- 登录页
- 记录列表页
- 关键词搜索
- 记录详情页
- 编辑 `internalNote`

## 7. 表单字段

### 7.1 个人直通卡

- 姓名
- 联系方式
- 项目名称
- 个人角色
- 项目一句话介绍
- 备注

### 7.2 团队直通卡

- 团队名称
- 主联系人姓名
- 主联系人联系方式
- 团队人数
- 项目名称
- 主联系人角色
- 项目一句话介绍
- 备注

## 8. 数据模型

系统使用一张主表 `passes`，同时承载个人和团队记录。

### 8.1 表结构

- `id`
- `type`
- `status`
- `name`
- `teamName`
- `contactName`
- `contactInfo`
- `role`
- `projectName`
- `projectSummary`
- `teamSize`
- `userNote`
- `internalNote`
- `createdAt`
- `updatedAt`

### 8.2 字段说明

- `id`：公开唯一 ID，建议使用不可预测字符串，如 `cuid` 或 `uuid`
- `type`：`individual` 或 `team`
- `status`：本期固定为 `active`
- `name`：个人记录时保存姓名
- `teamName`：团队记录时保存团队名称
- `contactName`：个人记录可与姓名相同，团队记录保存主联系人姓名
- `contactInfo`：联系方式
- `role`：个人角色或团队主联系人角色
- `projectName`：项目名称
- `projectSummary`：项目一句话介绍
- `teamSize`：团队人数，个人记录时可为空
- `userNote`：用户在表单中填写的备注
- `internalNote`：管理员内部备注，不公开展示

### 8.3 一张表的原因

- 这个系统规模较小
- 个人和团队在业务上都属于“直通卡记录”
- 列表搜索、详情展示和二维码生成逻辑都可以复用
- 比拆成两张表更简单，也更适合 MVP

## 9. Admin 认证设计

认证规则采用最小实现：

- 管理后台使用单一密码登录
- 密码存放在环境变量中
- 登录成功后写入带有效期的 session cookie
- 未登录访问 `/admin` 相关页面时跳转到 `/admin/login`

本期不做：

- 用户名密码体系
- 管理员账号管理
- 第三方登录
- 细粒度权限控制

## 10. 部署与运行约束

### 10.1 部署

- 生产部署平台：`Vercel`
- 数据库：`Neon Postgres`
- 环境变量在 `Vercel` 项目中配置

### 10.2 免费版前提

当前默认按低访问量、个人使用的 MVP 来设计：

- 访问量较低
- 管理员数量很少
- 数据规模有限
- 不追求复杂分析功能

如果后续活动规模扩大，再升级 `Vercel` 或 `Neon` 方案。

## 11. MVP 验收标准

以下能力全部可用，视为 MVP 完成：

- 首页可选择 `个人` 或 `团队`
- 个人表单可成功提交
- 团队表单可成功提交
- 提交后可生成第二个二维码
- 第二个二维码可打开对应详情页
- 详情页可正确展示对应记录
- `admin` 可通过密码登录
- `admin` 可查看记录列表
- `admin` 可搜索记录
- `admin` 可编辑并保存 `internalNote`
- 未登录用户不能访问 `admin`

## 12. 风险与边界

### 12.1 公开页信息暴露风险

公开详情页会暴露审核所需信息，因此要控制展示字段。当前为了线下审核效率，默认展示完整联系方式。这是一个有意识的取舍。

如果后续认为敏感度过高，可改为：

- 联系方式脱敏
- admin 内显示完整信息，公开页显示部分信息

### 12.2 链接猜测风险

如果 `id` 使用可预测值，可能被批量猜测。为避免这一点，`id` 必须使用不可预测字符串。

### 12.3 与报名系统的关系

本系统不负责最终报名，不与飞书表单自动联动。它的职责是：

- 收集直通卡对象信息
- 生成可扫描的唯一展示页
- 辅助人工审核
